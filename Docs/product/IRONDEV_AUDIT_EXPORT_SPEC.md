# IronDev Audit Export and Evidence Detail Specification

**Status:** Product and implementation contract

**Surface:** `/projects/{projectId}/library/audit`

## Purpose

Audit is the authoritative read-only history for project activity. This contract adds a bounded export and clearer evidence inspection without turning Audit into an authority surface or an analytics product.

The user should be able to:

- filter the project ledger;
- inspect one row without losing the filter context;
- follow a safe product evidence link;
- preview the exact export scope;
- download a backend-produced JSON package;
- understand truncation, generation time, schema version, and redaction boundaries.

## Product Boundary

Audit records and exports what happened. It does not:

- approve or refuse work;
- continue a workflow;
- apply or roll back source;
- replay an operation;
- reveal raw payload JSON, credentials, source content, prompts, completions, or private reasoning;
- claim compliance, safety, completeness, or forensic sufficiency;
- calculate charts, trends, risk scores, or team performance.

## Canonical Routes

```text
/projects/{projectId}/library/audit
/projects/{projectId}/library/audit/events/{ledgerId}
```

The event route is a client detail state backed by the row returned from the project ledger. Evidence links navigate to their authoritative product surface.

## Backend Export Contract

```http
GET /api/projects/{projectId}/audit/export
```

Supported query fields:

- `actor`
- `event`
- `workItemId`
- `fromUtc`
- `toUtc`
- `take`

`take` defaults to 250 and is capped at 250. The backend applies the same tenant, project-membership, date-range, actor, event, and Work Item rules as the visible ledger.

Response model:

```text
ProjectAuditExport
- SchemaVersion
- ProjectId
- ProjectName
- GeneratedUtc
- Filters
- ReturnedCount
- Take
- Truncated
- ItemsSha256
- Items
- Warnings
- Boundary
```

`Items` uses the existing redacted `AuditLedgerItem` contract. `ItemsSha256` binds the ordered exported rows and non-secret fields; it is evidence integrity metadata, not a signature or approval.

`Truncated` is true when the bounded result reaches `Take`. The API does not claim no later matching records exist.

## Export Boundary

The package is:

- JSON only;
- generated by the backend;
- project-scoped;
- tenant-authorized;
- bounded;
- non-secret;
- read-only;
- deterministic over the returned ordered items except for `GeneratedUtc`.

The browser downloads the returned package exactly. It must not reconstruct export rows from the DOM.

The filename is:

```text
irondev-audit-project-{projectId}-{yyyyMMdd-HHmmss}Z.json
```

## Export Workflow

Audit header actions:

```text
[Export current view]
```

Selecting export opens a preview containing:

- project name;
- active filters;
- maximum row count;
- returned row count after generation;
- truncation status;
- generated time;
- schema version;
- item hash;
- redaction and authority boundary.

Commands:

```text
[Generate export]
[Download JSON]
[Close]
```

`Download JSON` remains unavailable until generation succeeds. A failed generation produces no file and no success state.

Changing filters invalidates the prior preview package.

## Evidence Detail

Selecting an audit row opens a detail panel or canonical event route with:

- time;
- actor identity as recorded;
- project and Work Item reference;
- source;
- event;
- outcome;
- safe summary;
- correlation ID when present;
- evidence links;
- read-only boundary.

The detail view never renders raw event payloads. Unknown or unsafe external links are not opened.

## Link Safety

Allowed evidence targets are same-product relative paths beginning with `/` and not `//`.

Known product targets include:

- project Work Item;
- project Workshop session/channel;
- project document/version;
- project Members;
- project Governance/Audit;
- preserved read-only governance evidence routes.

Absolute URLs, script URLs, data URLs, and paths targeting another project are not rendered as active evidence links.

## Failure States

- `404`: project not found or outside selected tenant.
- `403`: current user cannot read the project ledger.
- `400`: invalid filter or date range.
- `Unavailable`: export could not be generated; retry performs the same read.
- `Empty`: valid package with zero rows and `Truncated = false`.

The client does not fall back to exporting visible table text.

## Required Tests

Backend:

- tenant and project isolation;
- project-membership refusal;
- date/filter validation parity with ledger;
- 250-row cap and truncation truth;
- stable ordered item hash;
- raw payload and secret marker exclusion;
- safe evidence-link preservation;
- missing project behavior.

Frontend:

- current filters reach export unchanged;
- preview before download;
- download disabled before successful generation;
- filter changes invalidate generated package;
- empty, failed, and truncated states;
- event detail and back behavior;
- unsafe/cross-project links are inert;
- no approve, continue, apply, rollback, replay, or mutation command;
- keyboard and 390px layout.

## Delivery Slices

1. `AUDIT-UX-0` - contract and information architecture.
2. `AUDIT-EXPORT-1` - backend project export package.
3. `AUDIT-UX-1` - export preview and download workflow.

## Acceptance Line

Audit can produce a bounded, non-secret project history package and guide a user to authoritative evidence without granting or replaying any consequential action.
